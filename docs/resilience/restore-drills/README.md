# Restore-drill evidence

> **Owner:** SRE Lead (with DPO for data-loss/compliance) | **Status:** Living evidence log
> This folder holds the **evidence records** for restore drills required by the backup-restore
> policy. A backup that is never restored is not a backup — these records prove the monthly
> verification actually happened and met its objectives.

## Convention

- One file per drill, named **`YYYY-MM-DD.md`** (UTC drill date).
- Records: what was restored, RTO achieved, integrity result, issues/follow-ups, owner.
- Generated by `scripts/restore/run_restore_drill.sh` (ADR-0082) — the orchestrator writes a
  filled template here. For an `--execute` drill, complete the `TODO` fields (RTO, integrity)
  after the integrity checks pass.
- Drives the monthly verification in `docs/resilience/backup-restore-policy.md` (§Backup
  verification & drills) and the review cadence in `docs/resilience/dr-plan.md`.

## How a drill is run

```bash
# Safe dry-run (no AWS mutations) — what CI runs and what you run locally to check tooling:
scripts/restore/run_restore_drill.sh --dry-run

# Real restore into throwaway targets (human-approved drill; requires AWS credentials):
scripts/restore/run_restore_drill.sh --execute
```

Per-store scripts (`restore_aurora_pitr.sh`, `restore_redis_snapshot.sh`,
`restore_kafka_snapshot.sh`) are parameterised by environment variables and also default to
dry-run. See each script's `--help`.

> The **scheduled** monthly CI drill (`backup-restore-verification.yml`) is **not yet wired** —
> it is DevOps-owned and gated on an RFC (ADR-0082 §Follow-up). Until then, drills are run
> manually and recorded here.

## Evidence template

```markdown
# Restore drill — YYYY-MM-DD

- **Date (UTC):** YYYY-MM-DDThh:mm:ssZ
- **Owner:** <name>
- **Mode:** execute | dry-run
- **Overall result:** PASS | FAIL
- **Trigger:** approved drill / scheduled workflow | dry-run (no AWS mutations)

## What was restored

| Store                        | Script                                      | Plan/restore result |
| ---------------------------- | ------------------------------------------- | ------------------- |
| Aurora PostgreSQL (PITR)     | `scripts/restore/restore_aurora_pitr.sh`    | PASS                |
| Redis (ElastiCache snapshot) | `scripts/restore/restore_redis_snapshot.sh` | PASS                |
| Kafka (MSK snapshot)         | `scripts/restore/restore_kafka_snapshot.sh` | PASS                |

## RTO achieved

- Wall-clock restore→integrity-pass vs per-service RTO (dr-plan.md / RB-002).

## Integrity result

- Row counts / key sampling / topic offsets against the restored throwaway target.

## Issues / follow-ups

- Any anomalies; file issues as needed.
```
