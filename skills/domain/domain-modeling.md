## Tactical DDD Patterns in This Codebase

### Building Blocks

| Pattern            | Definition                                                                    | When to use                                                            |
| ------------------ | ----------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| **Entity**         | Object with a stable identity that persists over time                         | Business objects that need lifecycle management (e.g., `DomainEntity`) |
| **Value Object**   | Immutable object defined by its attributes, no identity                       | Status codes, money amounts, measurements, tags                        |
| **Aggregate**      | Cluster of entities/value objects with a single root that enforces invariants | When consistency must be maintained across related objects             |
| **Domain Service** | Stateless operation that doesn't belong to any single entity                  | Cross-entity business rules, orchestration logic                       |
| **Repository**     | Abstraction over persistence — the only path to load/store aggregates         | Always — never access the DB directly from a service boundary          |
| **Domain Event**   | Immutable record that something happened in the domain                        | Cross-aggregate communication; triggering downstream workflows         |

### Reference: Java Domain Service Structure

```
services/domain-service/src/main/java/com/yourorg/domainservice/
├── domain/

│   └── EntityNotFoundException.java # Domain exception

│   ├── DomainEntityController.java # API adapter — delegates to domain service

    └── kafka/DomainEventProducer.java
```

## Entity Design Rules

1. **Assign identity at creation, never after.** Use UUIDs generated by the application layer, not by the database.
2. **Protect invariants in the aggregate root.** The root entity is the only entry point — no external code modifies child entities directly.
3. **Keep entities anemic-free.** Business logic belongs in the entity or domain service, not in the controller or worker.
4. **Avoid bidirectional references across aggregates.** Use IDs to reference other aggregates; load them via their own repository.

```java
// Good — invariant enforced in the aggregate root
public void activate() {
    if (this.status == EntityStatus.ARCHIVED) {
        throw new IllegalStateException("Cannot activate an archived entity");
    }
    this.status = EntityStatus.ACTIVE;
    this.updatedAt = Instant.now();
}

// Bad — status set from outside, invariant bypassed
entity.setStatus(EntityStatus.ACTIVE);
```

## Value Objects

Value objects are **immutable** and compared by value, not by reference.

```java
// Good value object
public record EntityStatus(String code, String label) {
    public static final EntityStatus ACTIVE = new EntityStatus("ACTIVE", "Active");
    public static final EntityStatus ARCHIVED = new EntityStatus("ARCHIVED", "Archived");
}

// Python equivalent
from enum import Enum
class EntityStatus(str, Enum):
    ACTIVE = "active"
    ARCHIVED = "archived"
```

**Rule:** If two instances with identical values are interchangeable, it is a value object. If they must be distinguishable even with identical values, it is an entity.

## Repository Pattern

Repositories hide persistence details. The domain layer defines the interface; the infrastructure layer implements it.

```java
// Domain layer — interface only (domain/DomainEntityRepository.java)
public interface DomainEntityRepository extends JpaRepository<DomainEntity, UUID> {
    Optional<DomainEntity> findByCorrelationId(String correlationId);
    List<DomainEntity> findByStatusAndCreatedAtBefore(EntityStatus status, Instant cutoff);
}

// Never in a domain service: EntityManager em; em.createNativeQuery(...)
```

```python
# Python — repository abstraction (see src/agents/hitl_store.py pattern)
class RequestStorePort(Protocol):
    async def get(self, request_id: str) -> RequestState | None: ...
    async def save(self, state: RequestState) -> None: ...

# Infrastructure implementations: RedisRequestStore, InMemoryRequestStore
```

**Rules:**

- Domain services depend only on the interface, never on a concrete implementation.
- Only aggregate roots have repositories. Child entities are loaded through the root.
- The repository is responsible for mapping between domain objects and persistence models.

## Domain Events

Publish domain events when something significant happens in the domain. Consumers handle downstream effects asynchronously.

```java
// Publishing a domain event after state change
public DomainEntity process(String correlationId, Map<String, Object> payload) {
    var entity = new DomainEntity(UUID.randomUUID(), correlationId, payload);
    repository.save(entity);
    eventProducer.publish(new EntityProcessedEvent(entity.getId(), entity.getCorrelationId()));
    return entity;
}
```

**Avro schema for domain events:** `infrastructure/message-broker/schema-registry/avro/`
**Topic definitions:** `services.yaml` (canonical service registry)

## Adding a New Domain Concept — Checklist

Before writing any code, answer:

- [ ] Is this an entity or a value object? (identity vs. equality)
- [ ] Which aggregate does it belong to?
- [ ] Does the aggregate root enforce all invariants for this concept?
- [ ] Does this require a new repository, or can it be loaded through an existing one?
- [ ] Does a state change need to publish a domain event?
- [ ] Does introducing this concept require a new Avro schema? (update `services.yaml` and `asyncapi.yaml`)
- [ ] Does any field contain PII? (apply `skills/privacy/pii.md`)
- [ ] Does this introduce a new architectural decision? (write an ADR)

## Anti-Patterns to Avoid

| Anti-pattern                     | Why it's harmful                                         | Correct approach                                |
| -------------------------------- | -------------------------------------------------------- | ----------------------------------------------- |
| Anemic domain model              | Business rules scattered across services and controllers | Push logic into entities and domain services    |
| DB entity as domain model        | Leaks persistence concerns into business logic           | Use separate domain objects and mappers         |
| Repository bypass                | Breaks invariants; allows partial aggregate states       | Always use the repository interface             |
| Direct cross-aggregate reference | Creates tight coupling; violates consistency boundaries  | Reference by ID; use domain events              |
| Fat service                      | Domain service becomes a transaction script              | Delegate to entities; service only orchestrates |
